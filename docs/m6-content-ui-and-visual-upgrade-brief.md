# M6 内容、UI 与视觉升级简报

## 1. 这份文档解决什么问题

在上一版 M6 规划里，方向更偏向“内容扩张与重复游玩性”。

但结合最新目标，当前项目真正要优先推进的，不只是继续补玩法内容，而是把下一阶段明确收束成三条主线：

1. 更多的人物特性、武器、地图
2. 更像正式游戏的 UI 表现，而不是主要依赖文字说明
3. 更真实的视觉表现，包括武器资产、角色视觉辨识度、场景资源表现

所以从这次开始，M6 不再只是“内容变多”，而是：

**内容扩张 + UI 改版 + 视觉资产接入**

---

## 2. 当前版本最明显的现状

基于现有工程结构，当前版本已经很清楚地暴露出三个现实情况：

### 2.1 内容结构已经成立，但池子还太小

当前已经有：

- 3 名角色
- 3 把武器
- 1 张主地图

这足以验证系统成立，但还不足以形成“内容厚度”。

### 2.2 UI 仍然明显是原型式文字界面

当前主菜单和 HUD 以程序动态创建的 `Label`、`OptionButton`、`PanelContainer` 为主。

玩家现在看到的，更多是：

- 文本说明
- 下拉框
- 文本预览

而不是：

- 角色卡片
- 武器图标
- 地图预览
- 清晰的选中反馈
- 更完整的开局和结算表现

### 2.3 视觉表现仍主要依赖程序绘制

目前项目中还没有正式的 `assets` 资源目录，战斗表现主要来自：

- `fighter.gd` 内的 `draw_rect`
- `projectile.gd` 内的 `draw_rect`
- `stage.gd` 内的大量程序绘制背景和平台

这意味着当前版本虽然能玩，但视觉上仍然很像技术原型，而不是内容逐渐成型的游戏版本。

---

## 3. M6 新的阶段定义

结合你的目标，M6 最合适的定义应该改成：

**第一轮产品化建设阶段**

它不是最终打磨，也不是联网化，而是把游戏从“稳定原型”推进到“看起来、玩起来都更像一个完整作品”的阶段。

M6 的核心不是只问：

- 有没有更多内容

而是进一步问：

- 这些内容有没有被视觉化地呈现出来
- 玩家能不能更舒服地理解和选择内容
- 游戏是不是开始具备真正的成品感

---

## 4. M6 的三条核心主线

## 4.1 主线一：内容扩张

这是第一优先级，因为它决定你们的核心卖点还能不能继续成长。

### 这条线要补什么

- 新角色
- 新武器
- 新地图

### 这一轮建议的最小目标

- 角色从 `3` 扩到 `4`
- 武器从 `3` 扩到 `4`
- 地图从 `1` 扩到 `3`

### 这条线的意义

只有当角色、武器、地图都扩起来之后：

- 组合系统才会真正有厚度
- 玩家才会开始对不同对局产生新鲜感
- UI 和视觉升级才有足够内容可展示

---

## 4.2 主线二：UI 表现升级

这是第二优先级，因为你已经明确不希望游戏一直停留在“文本驱动的原型 UI”。

### 当前 UI 的核心问题

- 选择界面是下拉框式，而不是内容卡片式
- 玩家理解角色/武器/地图差异主要靠读字
- 结果页、暂停页、开局提示仍偏功能化
- 训练 HUD 有信息，但还缺少更清晰的层级与视觉节奏

### 这条线要达成什么

- 主菜单从“表单式”变为“选择页式”
- 角色和武器拥有视觉卡片
- 地图选择拥有预览位
- 当前组合拥有更清楚的摘要展示
- 开局、结算、暂停、模式提示更有层次

### 这条线的意义

它直接决定玩家第一次看到游戏时，会不会觉得：

- 这是一个原型工具
或者
- 这是一个正在成型的游戏

---

## 4.3 主线三：视觉资产接入

这是第三优先级，但它不是可选项，而是必须正式立项的一条线。

因为你已经明确提出：

**希望游戏真的有武器资产，而不是继续只靠文本和基础图形表达。**

### 这条线要补什么

- 武器图标
- 武器局内表现资源
- 角色视觉表现资源
- 地图视觉资源
- 基础战斗特效资源

### 当前最重要的问题不是“画得够不够精”

而是：

- 工程里还没有正式资源目录
- 角色和武器数据库里还没有视觉资源字段
- 局内角色和武器还没有标准化挂点和表现入口
- 缺少“没有资产时用默认占位，有资产时自动切换”的接入机制

### 这条线的意义

它决定项目能不能从现在的“系统原型”过渡到真正可持续生产的内容型项目。

---

## 5. 下一阶段最合理的推进顺序

如果要考虑开发收益和风险，建议按下面顺序推进：

1. 先开内容线基础波次
   先补 `1 角色 + 1 武器 + 2 地图`

2. 同时开 UI 结构改版
   先把主菜单、选择页、结算页从文本原型升级成结构化视觉页面

3. 然后开资产接入底层
   建正式 `assets` 目录、资源字段、加载逻辑、占位回退机制

4. 再把第一批真实武器资源接进游戏
   优先接 `saber / hand_cannon / halberd`

5. 最后做一轮局内外整体表现收口
   包括 HUD、提示、地图展示、开局和结算演出

这个顺序的好处是：

- 先保证项目继续长内容
- 再保证内容能被更好地看见
- 最后让内容被真正视觉化落地

## 7. 对内容设计的直接建议

既然你明确想要“更多人物特性、武器、地图”，那下一波内容不建议只做数量增加，而是要做“结构补位”。

### 新角色建议

优先补当前阵容中缺失的风格：

- `trickster`：轻量、扰动、节奏破坏型
- `precision`：精确、中低机动、强判定惩罚型

### 新武器建议

优先补当前武器池中缺失的风格：

- `gauntlets`：超近身连段和压制
- `bow`：蓄力和线路控制
- `chakram`：回旋、路线规划、节奏型投射

### 新地图建议

建议 3 张地图各自承担不同定位：

- 标准竞技图
- 空战平台图
- 压边回场图

---

## 8. 对 UI 的直接建议

下一轮 UI 不建议只是“把文字摆得更整齐”，而应该直接从结构上改变：

- 用卡片替代单纯下拉框
- 用视觉选中态替代纯文本确认
- 用摘要区解释当前组合，而不是让玩家自己拼信息
- 用地图缩略预览位替代地图纯名字
- 用更明确的结果页替代单行结果文本

一句话说：

**要把 UI 从“开发工具界面”推进成“可展示的游戏界面”。**

---

## 9. 对视觉资产的直接建议

“真的有武器资产”这一条，建议不要晚做。

因为一旦继续纯程序绘制推进更多内容，后面再切资源化，会产生重复返工。

所以 M6 里最好尽早建立：

- `assets/weapons/`
- `assets/characters/`
- `assets/stages/`
- `assets/ui/`
- `assets/fx/`

同时让数据库和装配逻辑开始支持：

- 图标路径
- 局内贴图路径
- 预览图路径
- 缺失资源时的默认回退

---

## 10. 最后的阶段结论

如果按你现在的目标来定，M6 最准确的一句话应该是：

**M6 不是单纯加内容，而是让游戏同时变得更丰富、更好看、更像一个真正的游戏。**

接下来的优先级就很明确：

1. 扩角色、武器、地图
2. 重做主菜单和 HUD 的视觉结构
3. 建立真实资源资产接入管线
4. 把第一批武器与角色视觉真正接进战斗里
