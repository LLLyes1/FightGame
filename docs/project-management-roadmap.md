# 2D 平台格斗项目管理路线图

## 1. 文档目的

这份文档不替代玩法设计稿和技术规格书，而是补上 **项目推进层** 的内容：

- 当前项目做到哪一步了
- 还差哪些关键里程碑
- 任务应该怎么拆
- 近期开发优先级是什么
- 每个阶段的验收标准是什么

适用对象：

- 项目负责人 / PM
- 程序
- 策划
- 后续可能加入的 UI、美术、测试

## 2. 当前项目状态判断

基于当前 `docs` 和代码实现，项目已经不是从 0 开始，而是处于：

**“核心战斗原型已搭起，正在从技术原型过渡到可管理的 MVP 开发阶段。”**

### 2.1 已有成果

当前已经具备这些基础能力：

- 主菜单，可进入 `本地对战` / `训练模式`
- 1 张测试地图
- 2 个可玩的原型角色
- 基础移动、跳跃、空中移动
- 地面 / 空中可用的冲刺体系
- 轻攻击 / 重攻击
- 近战攻击判定
- 远程投射物攻击
- 受击、伤害累计、击飞
- 出界、复活、胜负结算
- 基础 HUD、暂停、重开、返回菜单
- 已有较完整的冲刺系统规格文档

### 2.2 当前最关键的缺口

虽然原型可玩，但从 PM 角度看，还存在 4 个核心缺口：

1. **装配玩法还没真正落地**
   当前代码里角色数据和攻击数据仍然绑定在同一个 `fighter_data` 中，对局生成也还是写死的两名角色，并没有真正实现“人物模块 + 武器模块自由组合”。

2. **缺少项目推进型文档**
   现在的文档偏设计说明与技术说明，缺里程碑、优先级、任务池、验收标准、提测清单。

3. **MVP 完整体验链路还没闭环**
   目前已经能打，但“选人 / 选武器 / 组合预览 / 开局确认 / 结算复盘”的产品链路还没有形成一个完整演示切片。

4. **测试和调优流程还没制度化**
   当前更像一次性实现，缺少针对手感、判定、回场、击飞、资源平衡的固定检查表。

### 2.3 当前阶段结论

当前最应该做的，不是继续横向加内容，而是：

**先把 MVP 的主路径做完整，把“自由搭配”这个项目卖点真正做出来。**

## 4. 项目里程碑规划

建议把当前项目拆成 4 个明确里程碑。

## M0. 当前状态：战斗技术原型

### 目标

验证移动、冲刺、攻击、击飞、出界循环成立。

### 退出标准

- 双人本地对战可运行
- 至少两种战斗风格可对打
- 可完成一整局并正常结算

### 状态

**基本完成**

---

## M1. 可演示 MVP

### 目标

把“能打”升级为“能演示、能说明核心卖点”的版本。

### 必须完成

- 选人流程
- 选武器流程
- 组合预览
- 基于组合生成战斗实体
- 1 套完整 UI 流程：主菜单 -> 选人/选武器 -> 对战 -> 结算 -> 返回
- 基础调试信息保留

### 退出标准

- 非开发者也能理解玩法入口
- 能清晰展示“人物决定怎么动，武器决定怎么打”
- 任意合法组合都能进入战斗并完成一局

### 优先级

**P0**

---

## M2. 垂直切片版本

### 目标

让项目从“功能成立”进入“体验成形”。

### 必须完成

- 2 个人物模块差异明显
- 2 把武器模块差异明显
- 至少 4 种合法组合可玩且有风格差
- 1 张更像正式关卡的地图
- 更清晰的命中反馈、受击反馈、结算反馈
- 训练模式信息更完整

### 退出标准

- 玩家能感知不同组合的明确差异
- 一局对战有基本节奏感和观赏性
- 核心玩法可用于对外展示或继续招人协作

### 优先级

**P1**

---

## M3. 扩展准备版

### 目标

为继续做内容和系统扩展打地基。

### 必须完成

- 人物数据和武器数据模块化
- 战斗参数可配置化
- 基础测试清单常态化
- 新增角色 / 武器的接入成本下降
- 代码与文档结构稳定

### 退出标准

- 新增一个人物或武器不需要大改底层
- 文档、代码、任务拆分能相互对应
- 具备继续扩内容或做平衡调优的基础

### 优先级

**P1-P2**

## 5. 推荐任务拆分方式

建议以后所有开发任务都按下面格式建卡，避免“做一个 XXX 系统”这种过大任务。

### 5.1 标准任务卡模板

| 字段 | 说明 |
| --- | --- |
| 任务标题 | 直接写可交付结果 |
| 目标 | 这件事解决什么问题 |
| 输入 | 依赖哪些文档、数据、界面、代码 |
| 输出 | 要交付什么 |
| 验收标准 | 什么情况下算完成 |
| 优先级 | `P0 / P1 / P2` |
| 预估 | `0.5d / 1d / 2d / 3d+` |
| 负责人 | 谁来做 |
| 风险 | 有哪些前置依赖或不确定性 |

### 5.2 拆任务的原则

- 一个任务最好只对应一个明确结果
- 一个任务尽量在 `0.5-2 天` 内可完成
- 设计决策和实现任务分开
- 数据结构调整和 UI 表现调整分开
- 新系统先做主路径，再做边角优化

## 6. 现阶段任务优先级池

## 6.1 P0：必须先做

### T1. 拆分人物模块与武器模块数据

目标：
把当前 `fighter_data` 中混在一起的移动能力和攻击能力拆开，为自由组合做准备。

验收标准：

- 存在独立的人物配置结构
- 存在独立的武器配置结构
- 战斗实体通过组合装配生成

### T2. 对局前装配流程

目标：
实现“选人物 -> 选武器 -> 确认组合 -> 开局”的最短产品链路。

验收标准：

- 双方都能完成选择
- 能看到组合预览
- 能正常进入对局

### T3. 组合生成逻辑接入战斗场景

目标：
让 `GameScene` 不再写死两名角色，而是读取装配结果生成对战单位。

验收标准：

- `versus` 模式使用选中的组合开局
- `training` 模式也可指定测试组合

### T4. MVP 主流程 UI 闭环

目标：
让主菜单、装配、开局、结算、返回形成完整流程。

验收标准：

- 无需改代码即可完整走一轮
- 失败路径和返回路径正常

## 6.2 P1：主路径完成后立刻跟进

### T5. 训练模式调试增强

目标：
让训练模式真正服务调手感和调平衡。

建议内容：

- 显示当前组合
- 显示冲刺状态
- 显示伤害、击飞、剩余跳跃 / 冲刺
- 显示招式名称与命中反馈

### T6. 战斗反馈打磨

目标：
增强命中感、受击感、出界感、结算感。

建议内容：

- 更明确的命中提示
- 受击闪白 / 停顿 / 音效接口
- 出界提示
- 结算文案强化

### T7. 地图从测试平台升级为演示平台

目标：
让地图从“碰撞用测试场”升级到“可展示玩法节奏”的垂直切片地图。

验收标准：

- 主平台 + 副平台布局服务追击和回场
- 摄像机边界和出界体验稳定

## 6.3 P2：可延后，但要纳入排期

### T8. 新角色 / 武器接入模板

目标：
把新内容接入步骤写标准化，降低后续扩内容成本。

### T9. 回归测试清单

目标：
形成固定的版本自测流程。

### T10. 数据调优面板或参数集中区

目标：
减少为了调手感而频繁改散落参数的成本。

## 7. 未来两周建议排期

如果按一个人主开发推进，建议先按下面节奏走。

| 时间 | 目标 | 关键任务 |
| --- | --- | --- |
| 第 1-2 天 | 方案定稿 | T1 数据拆分方案、装配数据结构、流程图 |
| 第 3-5 天 | 跑通主路径 | T2 选人选武器流程、T3 组合生成接入 |
| 第 6-7 天 | UI 闭环 | T4 菜单到结算链路打通 |
| 第 8-10 天 | 可玩性加强 | T5 训练模式调试增强、T6 基础反馈补强 |
| 第 11-14 天 | 演示稳定 | T7 地图调整、Bug 修复、M1 验收 |

如果是多人协作，建议并行切分：

- 程序 A：数据结构与战斗装配
- 程序 B / UI：菜单、选择、预览、结算链路
- 策划 / PM：组合规则、验收标准、测试清单

## 8. 阶段验收标准

为了避免“感觉差不多了”，建议 M1 验收统一按下面口径检查。

### 8.1 流程验收

- 能从主菜单进入装配流程
- 双方都能选人物和武器
- 能显示当前组合信息
- 能正常进入战斗
- 对局结束后能回到菜单

### 8.2 战斗验收

- 每种组合都能正常移动、跳跃、冲刺、攻击
- 近战和远程都可命中
- 受击、击飞、出界、复活正常
- 不出现明显软锁、黑屏、无法结算

### 8.3 卖点验收

- 玩家能明显感知“人物差异”
- 玩家能明显感知“武器差异”
- 玩家能理解组合系统是本项目核心亮点

---

## 9. 当前最推荐立刻开工的 3 件事

### 第一优先

**先做数据拆分和装配结构。**

这是后面选人选武器、组合预览、扩内容的总前提。

### 第二优先

**做完整的赛前流程。**

没有装配流程，当前项目的核心卖点还没有被玩家看到。

### 第三优先

**补测试清单。**

因为这个项目已经开始进入手感调优阶段，没有固定回归项会很容易越改越乱。

---

## 10. PM 结论

当前项目最重要的管理判断是：

**不要急着继续加角色、加武器、加地图。先把“人物 + 武器自由搭配”的主路径做完整，并用里程碑和任务卡把开发节奏固定下来。**

只要 M1 做完整，这个项目就会从“有潜力的战斗原型”变成“能被别人看懂、能继续协作开发的 MVP”。
